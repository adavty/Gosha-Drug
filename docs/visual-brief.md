# Gosha AI — визуальная концепция презентации JMLC

## Задача дизайна

За пять минут визуально перевести Gosha из широкой концепции «AI-платформа для учёбы» в понятный **надёжный AI-workflow общей памяти дедлайнов**. Дизайн должен сохранить узнаваемость ранней Gosha Box, но главным героем становится не фантазия о платформе, а работающий state transition.

## Что взять из Gosha Presa

- чёрно-графитовую основу и холодный синий свет;
- светящуюся синюю сферу как персонажа/маскота;
- крупную уверенную типографику;
- телефонные mockups и Telegram-native контекст;
- эмоциональность, которая отличает проект от типичного технического deck.

Использовать старые изображения только если у Алена есть право на их публикацию. Персональные аватары, имена и содержимое реальных чатов из старых mockups в конкурсный deck не переносить.

## Что не переносить

- название `Gosha Box` как обещание новой платформы-чата;
- широкие сценарии встреч, course-aware AI, поиска материалов и постоянной AI-плашки как реализованные;
- слайды с большим user-flow graph;
- длинные списки общих инсайтов без provenance;
- generic AI-art вместо доказательств текущего MVP;
- утверждения «сократили процесс», «реализовано в MVP» или research counts без подтверждения.

## Визуальная система

### Палитра

- Background: `#05070B` / `#090D14`.
- Surface: `#111722` с тонкой рамкой `#283344`.
- Primary electric blue: `#119CFF`.
- Glow/cyan: `#61D8FF`.
- Success: `#5FE3A1`.
- Warning: `#FFC857`.
- Critical: `#FF5D6C`.
- Magenta `#FF35D3` — только для одного контрастного акцента или blocked path, не как второй основной цвет.
- Text: `#F7F9FC`; secondary `#A8B3C4`.

### Типографика

- Один современный grotesk с кириллицей: Inter, Manrope или Golos Text.
- Заголовки 30–44 pt, основной текст 18–24 pt в 16:9 deck.
- Не более 35–45 видимых слов на большинстве слайдов.
- Один слайд — одна мысль; доказательства крупнее декоративных элементов.

### Графический язык

- Сфера Gosha показывает состояние: `listening`, `preview`, `confirmed`, `blocked`, а не просто украшает слайд.
- Действия отображаются как короткие state cards и соединяются одной линией.
- У implemented evidence — синяя маркировка; measured — зелёная; planned validation — пунктирная рамка; critical blocked — красная.
- На архитектуре визуально разделить probabilistic и deterministic зоны вертикальной границей.
- Использовать реальные screenshots текущего build; если нужен телефонный mockup, экран должен оставаться читаемым.

## Структура: 8 слайдов / 5 минут

### 1. Дедлайн потерялся в чате — 0:00–0:35

**Текст:** «Защита в следующий четверг» → через три дня: «а когда сдаём?»

**Визуал:** два фрагмента синтетического чата, между ними поток сообщений; маленький Gosha появляется в точке потери.

**Не показывать:** статистику без источника.

### 2. Gosha делает срок общей памятью — 0:35–1:10

**Текст:** «Один создаёт. Другой получает. Группа помнит.»

**Визуал:** пять состояний `phrase → preview → confirm → retrieval → reminder/recovery`.

**Статус:** implemented/product contract отмечаются отдельно.

### 3. Почему не закреп / LMS / календарь / ChatGPT — 1:10–1:35

**Визуал:** компактная матрица из четырёх alternatives. Не ставить зелёные галочки только у Gosha; показывать сильную сторону каждого и оставшийся gap.

**Тезис:** преимущество Gosha — гипотеза, которую должен проверить baseline/pilot.

### 4. 55 секунд продукта — 1:35–2:25

**Визуал:** screenshots release build в последовательности: preview, confirm, retrieval другим actor, correction/audit, blocked cross-chat.

**Требование:** release commit в footer. Если используется записанное видео, оно должно быть снято с того же build; пока записи нет, fallback — проверенные скриншоты и воспроизводимый CLI transcript.

### 5. AI понимает язык. Backend контролирует последствия — 2:25–3:05

**Визуал:** слева LLM `intent + slots`, справа validators/state/DB/scheduler; между ними узкий typed contract. Красный stop перед DB до confirmation.

**Подписи:** `no tools`, `no SQL`, `no chat selection`, `fallback available`.

### 6. Что уже доказано — 3:05–3:40

**Визуал:** три крупных факта текущего release и один failure example. Например: tests, synthetic set, measured metric; рядом явная плашка «не real users / не LLM quality», если применимо.

**Нельзя:** переносить planned Bronze thresholds в колонку results.

### 7. Как будет доказана ценность — 3:40–4:20

**Визуал:** `2 baseline weeks → 4 pilot weeks × 6 chats`; activation показывает двух разных участников. Ниже три исхода: Expand / Iterate / Stop.

**Крупно:** North Star `WAUC`; guardrail `0 critical incidents`.

### 8. Почему я и почему AI Product — 4:20–5:00

**Визуал:** одна линия `Контур → диплом → Т-Банк → Gosha → AI Talent Hub` с одним доказательством под каждой точкой.

**Финальная фраза:** «Хочу перейти от управляемого prototype к измеренному production AI-продукту».

## Footer и маркировка доказательств

На слайдах 4–7 мелким, но читаемым текстом:

- release commit;
- dataset/provider version на result slide;
- `synthetic/offline` рядом с соответствующей метрикой;
- `planned, not conducted` рядом с pilot;
- URL репозитория на финальном слайде.

## Анимация и motion

- Только функциональные переходы: появление preview, затем confirmation boundary, затем commit.
- Не использовать бесконечный glow/pulse, тяжёлые 3D-повороты и видео-фоны.
- В PDF каждый слайд должен быть самодостаточен; все ключевые состояния видны без анимации.

## Критерий визуального успеха

Без устного объяснения зритель за 20 секунд должен понять:

1. продукт не оценивает обучение и не является новой соцсетью;
2. он хранит и поддерживает общий дедлайн;
3. LLM не выполняет действие напрямую;
4. показанные результаты отделены от будущего пилота.
